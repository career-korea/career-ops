# 모드: oferta — 전체 A-G 평가

후보자가 공고(텍스트 또는 URL)를 붙여넣으면, 항상 7개 블록(A-F 평가 + G 진위성)을 제공한다:

## Step 0 — 아키타입 감지

공고를 6개 아키타입 중 하나로 분류한다(`_shared.md` 참조). 하이브리드면 가장 가까운 2개를 명시한다. 이것이 결정하는 것:
- 블록 B에서 우선할 증거 포인트
- 블록 E에서 요약을 재작성하는 방식
- 블록 F에서 준비할 STAR 스토리

## Block A — 역할 요약

표로:
- 감지된 아키타입
- 도메인 (platform/agentic/LLMOps/ML/enterprise)
- 기능 (build/consult/manage/deploy)
- 시니어리티
- 원격 (full/hybrid/onsite)
- 팀 규모 (언급되면)
- TL;DR 1문장

## Block B — CV 매칭

`cv.md`를 읽는다. 각 JD 요구사항을 CV의 정확한 줄에 매핑한 표를 만든다.

**아키타입에 맞춰:**
- FDE → 전달 속도와 고객 대면 증거 포인트 우선
- SA → 시스템 설계와 통합 우선
- PM → 제품 디스커버리와 지표 우선
- LLMOps → evals, 옵저버빌리티, 파이프라인 우선
- Agentic → 멀티 에이전트, HITL, 오케스트레이션 우선
- Transformation → 변화 관리, 도입, 스케일링 우선

각각에 대한 완화 전략과 함께 **갭(Gaps)** 섹션. 각 갭마다:
1. 하드 블로커인가 nice-to-have인가?
2. 후보자가 인접 경험을 증명할 수 있는가?
3. 이 갭을 커버하는 포트폴리오 프로젝트가 있는가?
4. 구체적 완화 계획 (커버레터 문구, 빠른 프로젝트 등)

## Block C — 레벨과 전략

1. JD에서 **감지된 레벨** vs **해당 아키타입에서 후보자의 자연스러운 레벨**
2. **"거짓 없이 시니어로 어필" 계획**: 아키타입에 맞춘 구체적 문구, 강조할 구체적 성과, 창업 경험을 강점으로 포지셔닝하는 법
3. **"다운레벨 제안 시" 계획**: 보상이 공정하면 수용, 6개월 리뷰 협상, 명확한 승진 기준

## Block D — 보상과 수요

WebSearch로:
- 해당 역할의 현재 급여 (Glassdoor, Levels.fyi, Blind)
- 회사의 보상 평판
- 역할의 수요 트렌드

데이터와 인용 출처가 있는 표. 데이터가 없으면 지어내지 말고 없다고 명시한다.

## Block E — 맞춤화 계획

| # | 섹션 | 현재 상태 | 제안 변경 | 이유 |
|---|---------|---------------|------------------|---------|
| 1 | Summary | ... | ... | ... |
| ... | ... | ... | ... | ... |

매칭을 극대화하기 위한 CV 변경 Top 5 + LinkedIn 변경 Top 5.

## Block F — 면접 계획

JD 요구사항에 매핑된 STAR+R 스토리 6~10개 (STAR + **Reflection**):

| # | JD 요구사항 | STAR+R 스토리 | S | T | A | R | Reflection |
|---|-----------------|-----------------|---|---|---|---|------------|

**Reflection** 컬럼은 무엇을 배웠는지 또는 무엇을 다르게 할지를 담는다. 이것이 시니어리티를 드러낸다 — 주니어는 무슨 일이 있었는지 설명하고, 시니어는 교훈을 추출한다.

**스토리 뱅크:** `interview-prep/story-bank.md`가 있으면, 이 스토리들이 이미 있는지 확인. 없으면 새것을 추가. 시간이 지나면 어떤 면접 질문에도 적응 가능한 재사용 마스터 스토리 5~10개의 뱅크가 쌓인다.

**아키타입에 따라 선택·프레이밍:**
- FDE → 전달 속도와 고객 대면 강조
- SA → 아키텍처 결정 강조
- PM → 디스커버리와 트레이드오프 강조
- LLMOps → 지표, evals, 프로덕션 하드닝 강조
- Agentic → 오케스트레이션, 오류 처리, HITL 강조
- Transformation → 도입, 조직 변화 강조

추가로 포함:
- 추천 케이스 스터디 1개 (그들의 프로젝트 중 무엇을 어떻게 제시할지)
- 위험 신호 질문과 답변법 (예: "왜 회사를 매각했나?", "직속 부하 팀이 있나?")

## Block G — 공고 진위성

이 공고가 실제로 진행 중인 채용인지 나타내는 신호를 분석한다. 사용자가 채용 프로세스로 이어질 가능성이 높은 기회에 노력을 우선하도록 돕는다.

**윤리적 프레이밍:** 고발이 아니라 관찰을 제시. 모든 신호에는 정당한 설명이 있다. 어떻게 비중을 둘지는 사용자가 결정.

### 분석할 신호 (순서대로):

**1. 공고 신선도** (Step 0에서 이미 캡처된 Playwright 스냅샷에서):
- 게시일 또는 "X일 전" — 페이지에서 추출
- Apply 버튼 상태 (활성 / 마감 / 없음 / 일반 페이지로 리디렉션)
- URL이 일반 채용 페이지로 리디렉션됐으면 기록

**2. 설명 품질** (JD 텍스트에서):
- 구체적 기술, 프레임워크, 도구를 명시하는가?
- 팀 규모, 보고 체계, 조직 맥락을 언급하는가?
- 요구사항이 현실적인가? (경력 연수 vs 기술의 역사)
- 첫 6~12개월의 명확한 범위가 있는가?
- 급여/보상이 언급되는가?
- JD에서 역할 특정 내용 vs 일반 보일러플레이트 비율은?
- 내부 모순이 있는가? (신입 직함 + staff 요구사항 등)

**3. 회사 채용 신호** (WebSearch 2~3개 쿼리, 블록 D 리서치와 결합):
- 검색: `"{company}" layoffs {year}` — 날짜, 규모, 부서 기록
- 검색: `"{company}" hiring freeze {year}` — 발표 기록
- 정리해고가 발견되면: 이 역할과 같은 부서인가?

**4. 재게시 감지** (scan-history.tsv에서):
- 회사 + 유사 역할명이 다른 URL로 이전에 나타났는지 확인
- 몇 번, 어느 기간에 걸쳐 나타났는지 기록

**5. 역할 시장 맥락** (정성적, 추가 쿼리 없음):
- 보통 4~6주에 채워지는 흔한 역할인가?
- 이 회사의 비즈니스에 이 역할이 말이 되는가?
- 합법적으로 채우는 데 더 오래 걸리는 시니어리티인가?

### 출력 형식:

**평가:** 세 등급 중 하나:
- **High Confidence** — 다수 신호가 실제 진행 중 채용을 시사
- **Proceed with Caution** — 주목할 가치 있는 혼재 신호
- **Suspicious** — 다수 유령 공고 지표, 시간 투자 전 조사

**신호 표:** 관찰된 각 신호와 그 발견 및 가중치 (Positive / Neutral / Concerning).

**맥락 메모:** 잠재적으로 우려되는 신호를 설명하는 단서 (틈새 역할, 공공 일자리, 상시 채용 등).

### 엣지 케이스 처리:
- **공공/학계 공고:** 더 긴 일정이 표준. 임계값 조정 (60~90일이 정상).
- **상시/연속 채용 공고:** JD가 명시적으로 "ongoing" 또는 "rolling"이라 하면 맥락으로 기록 — 이것은 유령 공고가 아니라 파이프라인 역할.
- **틈새/임원 역할:** Staff+, VP, Director, 또는 고도로 전문화된 역할은 합법적으로 몇 달간 열려 있음. 게시 기간 임계값을 그에 맞게 조정.
- **스타트업 / 프리레비뉴:** 초기 단계 회사는 역할이 진짜 미정이라 JD가 모호할 수 있음. 설명 모호함을 덜 무겁게 가중.
- **날짜 없음:** 공고 게시 기간을 알 수 없고 다른 신호도 우려되지 않으면, 데이터가 제한적이었다는 메모와 함께 "Proceed with Caution"을 기본값으로. 증거 없이 절대 "Suspicious"를 기본값으로 하지 않음.
- **리크루터 소싱 (공개 공고 없음):** 신선도 신호 없음. 활발한 리크루터 접촉 자체가 긍정적 진위성 신호임을 기록.

## 평가 후

블록 A-G 생성 후 **항상**:

### 1. report .md 저장

전체 평가를 `reports/{###}-{company-slug}-{YYYY-MM-DD}.md`에 저장.

- `{###}` = 다음 순번 (3자리, 0 패딩)
- `{company-slug}` = 소문자 회사명, 공백 없이 (하이픈 사용)
- `{YYYY-MM-DD}` = 현재 날짜

**리포트 형식:**

```markdown
# 평가: {회사} — {역할}

**Date:** {YYYY-MM-DD}
**URL:**
**Archetype:** {감지된 것}
**Score:** {X/5}
**Legitimacy:** {High Confidence | Proceed with Caution | Suspicious}
**PDF:** {경로 또는 pending}

---

## A) 역할 요약
(블록 A 전체 내용)

## B) CV 매칭
(블록 B 전체 내용)

## C) 레벨과 전략
(블록 C 전체 내용)

## D) 보상과 수요
(블록 D 전체 내용)

## E) 맞춤화 계획
(블록 E 전체 내용)

## F) 면접 계획
(블록 F 전체 내용)

## G) 공고 진위성
(블록 G 전체 내용)

## H) 지원서 답변 초안
(점수 >= 4.5일 때만 — 지원서 양식용 답변 초안)

---

## 추출된 키워드
(ATS 최적화를 위한 JD의 키워드 15~20개 목록)
```

### 2. 트래커에 기록

`data/applications.md`에 **항상** 기록:
- 다음 순번
- 현재 날짜
- 회사
- 역할
- Score: 매칭 평균 (1~5)
- 상태: `Evaluated`
- PDF: ❌ (또는 auto-pipeline이 PDF를 생성했으면 ✅)
- Report: report .md에 대한 상대 링크 (예: `[001](reports/001-company-2026-01-01.md)`)

**트래커 형식:**

```markdown
| # | 날짜 | 회사 | 역할 | Score | 상태 | PDF | Report |
```
