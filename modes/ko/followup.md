# 모드: followup — 후속 연락 주기 트래커

## 목적

진행 중인 지원의 후속 연락(follow-up) 주기를 추적한다. 기한이 지난 후속 연락을 표시하고, 메모에서 연락처를 추출하며, 리포트 컨텍스트로 맞춤 후속 이메일/LinkedIn 초안을 생성한다.

## 입력

- `data/applications.md` — 지원 트래커
- `data/follow-ups.md` — 후속 연락 이력 (첫 사용 시 생성)
- `reports/` — 평가 리포트 (초안 컨텍스트용)
- `config/profile.yml` — 사용자 프로필 (이름, 신원)
- `cv.md` — 초안의 증거 포인트용 CV

## Step 1 — 주기 스크립트 실행

실행:

```bash
node followup-cadence.mjs
```

JSON 출력을 파싱한다. 포함 내용:

| 키 | 내용 |
|-----|----------|
| `metadata` | 분석일, 총 추적 수, 실행 가능 수, overdue/urgent/cold/waiting 카운트 |
| `entries` | 지원별: 회사, 역할, 상태, 지원 후 경과일, 후속 횟수, 긴급도, 다음 후속일, 추출된 연락처, 리포트 경로 |
| `cadenceConfig` | 주기 규칙 (applied: 7일, responded: 3일, interview: 1일) |

실행 가능한 항목이 없으면 사용자에게 알린다:
> "후속 연락할 진행 중 지원이 없습니다. `/career-ops`로 먼저 몇 군데 지원하고, 시간이 지나면 다시 오세요."

## Step 2 — 대시보드 표시

긴급도 순(urgent > overdue > waiting > cold)으로 정렬한 주기 대시보드 표시:

```
후속 연락 주기 대시보드 — {date}
{N}개 지원 추적, {N}개 실행 가능

| # | 회사 | 역할 | 상태 | 경과일 | 후속 횟수 | 다음 | 긴급도 | 연락처 |
```

시각 표시 사용:
- **URGENT** — 24시간 내 응답 (회사가 답장함)
- **OVERDUE** — 후속 연락 기한 지남
- **waiting (X일)** — 정상, 후속 예정
- **COLD** — 2회 이상 후속 발송, 종료 권장

## Step 3 — 후속 연락 초안 생성

**overdue** 또는 **urgent** 항목에 대해서만:

1. 연결된 리포트(JSON의 `reportPath`)를 읽어 회사 컨텍스트 파악
2. `cv.md`에서 증거 포인트 읽기
3. `config/profile.yml`에서 후보자 이름과 신원 읽기

### 이메일 후속 프레임워크 (첫 후속, followupCount == 0)

3~4문장 이메일 생성:

1. **문장 1:** 구체적 역할 + 지원 시점 언급. 구체적으로 — 회사명과 역할명 명시.
2. **문장 2:** 리포트 블록 B 매칭이나 cv.md의 증거 포인트에서 구체적 가치 하나. 가능하면 정량화.
3. **문장 3:** 부드러운 요청 + 가능 시간. 구체적 시간대 제안 ("이번 주" 또는 "다음 화요일").
4. **문장 4 (선택):** 관련 최근 프로젝트나 성과를 간단히 언급.

**규칙:**
- 전문적이되 따뜻하게, 절박하지 않게
- "그냥 확인차", "후속 차", "안부차", "다시 연락" 같은 표현 **절대 금지**
- 요청이 아니라 가치를 먼저
- 그 회사에 특정한 무언가 인용 (리포트 블록 A에서)
- 150단어 이하 유지
- 제목 줄 포함
- `config/profile.yml`의 후보자 이름 사용

**예시 톤:**
> 제목: Re: Senior PHP/Laravel Developer — IxDF
>
> 안녕하세요 [담당자명 또는 "팀"],
>
> 4월 7일에 Senior PHP/Laravel Developer 역할에 지원했습니다. 제 프로덕션 Laravel 앱(Barbeiro.app — 모델 120개, API 엔드포인트 315개, 전체 테스트 스위트)이 공고에 설명된 TDD 중심 문화와 매우 닮아 있어 공유드립니다.
>
> 제 15년 PHP 경험과 실전 AI 도구 워크플로우가 IxDF 플랫폼에 어떻게 기여할 수 있을지 논의하고 싶습니다. 이번 주 중 짧게 대화 가능한 시간이 있을까요?
>
> 감사합니다,
> [이름]

### LinkedIn 후속 (이메일 연락처를 못 찾은 경우)

contacto 프레임워크 재사용: 3문장, 최대 300자.
- 회사 특정 후크 → 증거 포인트 → 부드러운 요청
- 적합한 사람을 먼저 찾도록 `/career-ops contacto {company}` 실행 제안

### 두 번째 후속 (followupCount == 1)

첫 번째보다 짧게 (2~3문장). **새로운 앵글**로:
- 관련 인사이트, 기사, 프로젝트 업데이트 공유
- 첫 후속 내용 반복 금지
- 여전히 역할을 구체적으로 언급

### 콜드 지원 (followupCount >= 2)

또 다른 후속을 생성하지 말 것. 대신 제안:
> "이 지원은 응답 없이 {N}회 후속 연락을 했습니다. 고려:
> - 역할이 채워진 것 같으면 상태를 `Discarded`로 업데이트
> - `/career-ops contacto`로 다른 연락처 시도
> - `Applied` 상태 유지하되 우선순위 낮추기"

## Step 4 — 초안 제시

각 초안에 대해 표시:

```
## 후속: {회사} — {역할} (#{num})

**받는 사람:** {email 또는 "연락처 없음 — 먼저 `/career-ops contacto` 실행"}
**제목:** {제목 줄}
**지원 후 경과일:** {N}
**발송한 후속:** {N}
**채널:** Email / LinkedIn

{초안 텍스트}
```

## Step 5 — 후속 연락 기록

사용자가 검토 후 후속 연락을 보냈다고 하면 기록한다:

1. `data/follow-ups.md`가 없으면 생성:
   ```markdown
   # 후속 연락 이력

   | # | App# | 날짜 | 회사 | 역할 | 채널 | 연락처 | 메모 |
   |---|------|------|---------|------|---------|---------|-------|
   ```

2. 다음 행 추가:
   - `#` = 후속 테이블의 다음 순번
   - `App#` = 트래커의 지원 번호
   - `날짜` = 오늘 날짜
   - `회사` = 회사명
   - `역할` = 역할명
   - `채널` = Email / LinkedIn / Other
   - `연락처` = 발송 대상
   - `메모` = 간단한 메모 (예: "첫 후속, Barbeiro.app 인용")

3. 선택적으로 `data/applications.md`의 메모 컬럼을 "후속 {N} 발송 {YYYY-MM-DD}"로 업데이트

**중요:** 사용자가 실제로 보냈다고 확인한 후속만 기록한다. 초안을 발송한 것으로 절대 기록하지 않는다.

## Step 6 — 요약

모든 초안을 보여준 후 요약:

> **후속 대시보드** ({date})
> - {N}개 지원 추적 중
> - {N}개 overdue — 위에 초안 생성됨
> - {N}개 urgent — 오늘 응답
> - {N}개 waiting — 다음 후속일 표시됨
> - {N}개 cold — 종료 고려
>
> 위 초안을 검토하고 어느 것을 보냈는지 알려주시면 기록하겠습니다.

## 주기 규칙 참조

| 상태 | 첫 후속 | 이후 | 최대 시도 |
|--------|----------------|------------|-------------|
| Applied | 지원 후 7일 | 7일마다 | 2회 (이후 cold 표시) |
| Responded | 1일 (긴급 답장) | 3일마다 | 제한 없음 |
| Interview | 1일 후 (감사 인사) | 3일마다 | 제한 없음 |

이 기본값은 `node followup-cadence.mjs --applied-days N`으로 재정의할 수 있다.
